# Release Automation Design

**Date:** 2026-03-27

## Goal

Add a low-maintenance release path for the `lifeos` npm CLI package. Pushing a `vX.Y.Z` tag should verify the project, publish the package to npm, create a GitHub Release, and attach the packed npm artifact.

## Constraints

- The package is distributed primarily through npm and `npx lifeos`.
- The repository already has CI but no release workflow.
- Core release rules should live in local scripts, not only in workflow YAML.
- GitHub Release should provide source snapshots plus an uploaded `npm pack` tarball.
- No multi-platform standalone binaries or bundled runtimes in this iteration.

## Chosen Approach

Use a single GitHub Actions workflow triggered by version tags and optional manual dispatch.

The workflow will:

1. Check out the tagged commit.
2. Install dependencies.
3. Verify the tag version matches `package.json`.
4. Run typecheck, lint, tests, and build.
5. Generate an npm tarball with `npm pack`.
6. Publish the package to npm.
7. Create a GitHub Release for the tag and upload the tarball artifact.

## Local Release Helpers

Add small Node-based helper scripts under `scripts/release/` so the critical release behavior is testable outside GitHub Actions:

- `check-version.mjs`: validates the current Git ref tag against `package.json`.
- `pack.mjs`: runs `npm pack`, captures the produced tarball name, and prints it for workflow consumption.

These scripts keep version validation and packaging logic in the repository instead of embedding shell-heavy logic in YAML.

## Workflow Shape

Use one workflow file, `release.yml`, with:

- `push.tags: ['v*']`
- `workflow_dispatch`
- a single release job on Ubuntu
- `contents: write` and `id-token: write`

`id-token: write` allows trusted publishing to npm if the repository enables npm provenance/trusted publishing. If that is not configured yet, the job can still use `NODE_AUTH_TOKEN`.

## Versioning Rules

- Git tags must use the `vX.Y.Z` format.
- The tag version must exactly equal `package.json.version`.
- A mismatch must fail before any publish or release step.

## Testing Strategy

Add tests for the local release helpers:

- matching tag passes
- mismatched tag fails
- missing tag input fails
- tarball name extraction works for `npm pack` output

Workflow YAML itself will not get direct unit tests; its correctness is anchored by the helper-script tests and repository CI.

## Out of Scope

- GitHub Release notes generation beyond a basic autogenerated body
- changelog automation
- semantic-release / changesets / release-please
- platform-specific standalone binaries
